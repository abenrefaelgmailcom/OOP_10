הסבר מפורט – שאלה 1: Hash + Equality עם set ו־dict
רקע קצר אבל קריטי

set ו־dict בנויים על Hash Table.

כשהכנסת אובייקט ל־set או השתמשת בו כמפתח ב־dict, פייתון עושה בערך את זה:

קורא ל־__hash__() כדי למצוא “תא” מתאים בטבלה.

אם יש כבר משהו באותו תא (collision), הוא בודק עם __eq__() האם זה “אותו” מפתח/אותו איבר.

החוזה (contract) החשוב:

אם a == b אז חייב להיות ש־hash(a) == hash(b).

אבל ההפך לא חייב להיות נכון (יכולים להיות same hash ועדיין לא שווים).

פה בתרגיל בכוונה “שוברים” את ההיגיון התקין כדי להבין מה קורה פנימית.

Version A – __hash__ קבוע ו־__eq__ תמיד True
מה יקרה ב־set?
s.add(obj_a)
s.add(obj_b)


לשניהם hash = 999 → הם נופלים לאותו מקום בטבלה.

עכשיו פייתון בודק obj_a == obj_b.

אצל AlwaysEqual, __eq__ תמיד מחזיר True (לאותו סוג), אז פייתון אומר:
✅ “האיבר כבר קיים” → לא מוסיף עוד אחד.

תשובה:

כמה פריטים יהיו ב־set?
✅ 1

למה?
כי ברגע שה־hash התנגש, __eq__ אמר שהם שווים → set שומר ייחודיות לפי “שוויון”.

מה יקרה ב־dict?
d[obj_a] = "apple"
d[obj_b] = "banana"


שוב: אותו hash → אותו “אזור” בטבלה.

פייתון בודק אם obj_b הוא אותו מפתח כמו obj_a לפי __eq__.

__eq__ מחזיר True → פייתון מתייחס לזה כאל אותו מפתח.

ולכן ההשמה השנייה דורסה את הערך הראשון.

תשובות:

כמה פריטים יהיו ב־dict?
✅ 1

מה יחזיר d[obj_a]?
✅ "banana"

האם ההכנסה השנייה דרסה את הראשונה?
✅ כן. כי המפתח נחשב “אותו מפתח” (לפי __eq__).

Version B – __hash__ קבוע ו־__eq__ תמיד False
מה יקרה ב־set?

לשניהם hash=999 → collision.

פייתון בודק obj_a == obj_b.

NeverEqual.__eq__ תמיד False → פייתון מחליט:
✅ “זה לא אותו איבר” → מוסיף גם את השני.

תשובה:

כמה פריטים יהיו ב־set?
✅ 2

למה?
כי למרות אותו hash, הם לא שווים, ולכן set מאפשר לשניהם להישאר (collision נפתר בעזרת בדיקות __eq__).

מה יקרה ב־dict?

d[obj_a] = "cat" נכנס.

d[obj_b] = "dog":

collision עם אותו hash,

בדיקת שוויון מחזירה False,

פייתון רואה בזה מפתח חדש,

ולכן שומר גם את השני.

תשובות:

כמה פריטים יהיו ב־dict?
✅ 2

מה יחזיר d[obj_a] עכשיו?
✅ "cat" (לא נדרס)

הבדל מרכזי בין Version A ל־B:

ב־A, “כולם שווים” → set/dict מתנהגים כאילו כל המפתחות אותו דבר → דריסה ואיבר יחיד.

ב־B, “אף אחד לא שווה” → set/dict שומרים את כולם בנפרד גם אם אותו hash.

מה מתקדם פה ב־OOP לעומת ההתחלה?

זה בדיוק “OOP מתקדם” כי אתה נכנס ל:

Dunder methods / Magic methods: __hash__, __eq__ – אלו “hooks” שמחברים את האובייקט שלך למנוע של פייתון.

שינוי התנהגות של מבני נתונים (set, dict) ע"י הגדרה של שוויון/האשהינג.

הבנת חוזים (contracts): אם אתה מגדיר __eq__, אתה חייב לחשוב על __hash__ (ואם האובייקט mutable – זה מסוכן).

הנדסת זהות מול שוויון:

is / identity (מי זה בזיכרון)

== / equality (מה המשמעות שאתה מגדיר)

זה מעבר חד מהתחלה של OOP (“יש לי class ושדות”) ל־OOP שמגדיר איך האובייקט משתלב בתוך המערכת של פייתון.

הסבר מפורט – שאלה 2: *args בדיקת כפילויות
מה זה *args?

*args מאפשר לפונקציה לקבל מספר לא ידוע של ארגומנטים מיקומיים:

has_duplicates(1, 2, 3, "a", True)


בפנים זה מתקבל כטופל (tuple) בשם args.

איך פתרתי ולמה זה נכון?

השתמשתי ב־set בשם seen:

סט נותן בדיקת שייכות מהירה מאוד (בדרך כלל O(1)).

עוברים איבר איבר:

אם כבר ראינו אותו → יש כפילות → True

אחרת מוסיפים ל־seen

למה זה “מתקדם”?

כי זה מחבר בין:

*args (פונקציה גמישה),

Hashing של ערכים,

שימוש ב־set כדי לייעל זמן ריצה.

בונוס find_duplicates:

שני סטים: seen ו־dupes.

כל מה שמופיע פעם שנייה נכנס ל־dupes.

הסבר מפורט – שאלה 3: **kwargs סינון מפתחות לפי prefix
מה זה **kwargs?

**kwargs מאפשר לקבל מספר לא ידוע של ארגומנטים בשם:

pick_keys(name="Dana", is_admin=True, age=20)


בפנים זה מתקבל כ־dict.

איך פתרתי ולמה?

יצרתי dict חדש result (כדי לא לשנות את המקור).

עברתי על kwargs.items().

בדקתי k.startswith(prefix).

שמרתי רק את אלה שמתאימים.

הבונוס שבניתי:

prefix="is_" כפרמטר — זה הופך את הפונקציה ליותר כללית.

למה זה מתקדם ב־OOP/פייתון?

כי זה עובר מהתחלה (“יש פונקציה שמקבלת פרמטרים קבועים”) ל:

API design גמיש,

“תכנות לפי קונבנציות” (prefix),

שימוש ב־dict ככלי לבניית תצורה/פילטרים כמו שעושים בפרויקטים אמיתיים.
